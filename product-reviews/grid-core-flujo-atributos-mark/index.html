Grid-Core · Product Review

Flujo de Atributos MARK

Reimplementación del flujo de atributos MARK para permitir su explotación libre como tag lógico en filtros de parrilla.

Contexto y motivación
Por qué se necesita este cambio y de dónde viene la demanda

A partir de una petición de negocio para identificar ciertos productos denominados Base de Venta (BdV) — productos con alta rotación tanto en tienda física como online — con el fin de crear secciones personalizadas y segmentadas a nivel de mercado, desde el equipo de Grid-Core se realizó un análisis conjunto con PACMAN y Arquitectura. La conclusión fue que la solución más correcta y sostenible a largo plazo era etiquetar estos productos mediante atributos de tipo MARK que indicasen su condición de producto BdV.

Sin embargo, debido a los plazos requeridos por negocio, esta solución no era factible en el corto plazo. Como medida temporal se decidió implementar un KPI que permitiese identificar estos productos a partir de información publicada en Snowflake.

En paralelo, desde el equipo de Grid-Core se diseñó e implementó el flujo definitivo basado en atributos MARK que resuelve este caso de uso de forma genérica, dando cabida a necesidades similares que puedan surgir en el futuro.

Solución técnica
El nuevo flujo y cómo se modela el atributo

La reimplementación elimina los comportamientos fijos asociados a los subtipos de MARK y expone las marcas como un nuevo tag lógico marks, por el que se podrán aplicar condiciones y filtros tanto en la query del grid como en la de las secciones.

Estructura del atributo MARK

📄 Modelo de datos recibido

El atributo llega desde el sistema origen con la siguiente estructura. El campo subtype actúa como identificador semántico de la marca (ej. BDV, STYLING_INFO).

JSON
{
  "type": { "value": "Mark" },
  "subtype": { "value": "BDV" },
  "identifier": { "value": "BDV" },
  "values": [],
  "properties": {}
}

Nuevo tag lógico

🔍 Tag marks en las queries

Se crea el tag lógico marks que permite filtrar por el subtype del atributo MARK en cualquier query de grid o sección. El siguiente ejemplo muestra su uso combinado con otros filtros habituales:

Query DSL
NORTH AND (woman OR (man AND Unisex))
AND marks:BDV OR marks:BASESTYLE     // filtra por mark
AND (NOT ((accessories) OR (perfumes))
     AND ((*:* AND NOT availability_status:COMING_SOON)
     AND (NOT(price_cut:[0.01 TO 1] OR labels:special_price))))
AND NOT (promotions:SALES**)

Casos de uso que deben funcionar con el nuevo flujo
1 Mark de tipo STYLING_INFO — Estilismo base de venta

Migración y coexistencia
Impacto en equipos dependientes y plan de transición

Ambos flujos — el antiguo con comportamientos fijos y el nuevo con el tag lógico marks — deberán convivir durante un período hasta completar la migración. Durante ese tiempo el sistema debe ser capaz de procesar los dos modelos sin conflicto.

📢 Impacto en PACMAN

El equipo de PACMAN deberá adaptar su integración para pasar a usar el nuevo campo de configuración al seleccionar estilismos. El campo anterior usado en el DLS (stylings:BASESTYLE) deberá sustituirse progresivamente.

Antes DLS con stylings:BASESTYLE
Ahora Campo mocacoOutfitSelectionSettings.customizations para seleccionar el estilismo, lo que permite usar un DLS del tipo marks:BASESTYLE
🔒 Contratos de API no afectados

Este cambio no altera los contratos existentes de los endpoints principales. Los siguientes elementos permanecen sin cambios:

✓ buildMiniGrid — sin cambios en el contrato
✓ getGridFilters — sin cambios en el contrato
✓ API de visibilidad — no afectada
⚠️

Acción requerida: Comunicar a PACMAN que deben migrar su configuración a mocacoOutfitSelectionSettings.customizations para hacer uso del nuevo tag lógico marks antes de que el flujo antiguo sea retirado.

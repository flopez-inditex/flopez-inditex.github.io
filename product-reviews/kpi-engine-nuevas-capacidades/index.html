KPI-Engine · Product Review

Nuevas capacidades y flexibilidad en las métricas

Rediseño del modelo de definición de KPIs para habilitar un catálogo unificado, expresivo y autónomo: nuevo contrato de API, filtros avanzados y migración de todas las métricas existentes al nuevo modelo.

Contexto y motivación
Por qué el modelo actual limita la evolución y qué problema resuelve este conjunto de cambios
01
Modelo actual — Fragmentación
KPIEngine expone métricas a través de un modelo basado en variables individuales, fórmulas y proyecciones. Cada métrica requiere configuración separada y el modelo no permite declarar de forma compacta conjuntos de métricas que comparten fuente, ventana temporal y dimensiones de agrupación.
02
Consecuencia — Fricción operativa
La creación de nuevas métricas exige configurar manualmente múltiples objetos interdependientes. Los consumidores no pueden definir ni modificar métricas de forma autónoma a través del API, lo que ralentiza la incorporación de nuevas necesidades de negocio.
03
Catalizador — Spike KPIENGINE-1127
El análisis de nuevas capacidades identificó la necesidad de un contrato de API unificado que permita definir métricas declarativamente: fuente, ventana temporal, frecuencia de refresco, dimensiones, filtros y operadores de agregación, todo en un único objeto.
04
Solución — Nuevo modelo + migración total
Se diseña un nuevo contrato de API que coexiste con el modelo anterior durante la transición. Las métricas existentes (ventas, add-to-cart, stock, ratios de conversión) se migran progresivamente. El modelo antiguo queda deprecated una vez completada la migración.
Nuevo contrato de API para definición de métricas
Un único objeto declarativo reemplaza la configuración fragmentada actual — KPIENGINE-1152 KPIENGINE-1153

El nuevo modelo permite que un consumidor declare en una única definición todo lo necesario para calcular y consultar un conjunto de métricas: fuente de datos, ventana temporal, granularidad, filtros y las métricas concretas a computar. El motor se encarga de planificar y ejecutar el cálculo de forma autónoma.

JSON — Ejemplo de definición de métrica 
{
  "id":          "sales_24h_store_component",
  "description": "Sales metrics last 24h by store and commercial component",
  "source":      "SALES",
  "window":      { "unit": "HOURS", "value": 24 },
  "schedule":    "5m",
  "aggregators": [ "STORE_ID", "COMMERCIAL_COMPONENT_ID" ],
  "filters":     [
    {
      "aggregator": "CATEGORY_ID",
      "operator":   "NOT_IN",
      "values":     [123, 234, 345]
    }
  ],
  "metrics": [
    { "operator": "SUM", "column": "units * price", "name": "sa24stcc" }
  ]
}
Campo Tipo Req. Descripción
idstringSíIdentificador único de la métrica
sourceenumSíFuente de datos: SALES, RETURNS, ADD_TO_CART…
window.unitenumSíUnidad temporal: HOURS, DAYS, WEEKS. Puede ser nulo para métricas de tipo snapshot (ej. stock)
window.valueintegerSíNúmero de unidades de la ventana. Entero positivo
schedulestringSíFrecuencia de recálculo: 5m, 1h, 1d
aggregatorsstring[]SíDimensiones de agrupación del resultado: STORE_ID, COMMERCIAL_COMPONENT_ID, PRODUCT_COLOR_ID…
filtersFilter[]NoCondiciones de filtrado antes de agregar. Operadores: IN, NOT_IN, NULL, NOT_NULL
metrics[].operatorenumSíFunción de agregación: SUM, AVG, COUNT, MIN, MAX
metrics[].columnstringSíColumna o expresión aritmética sobre la que se aplica el operador. Ej: units * amount
metrics[].namestringSíNombre único del resultado dentro de la métrica
💡
Integración interna transparente: el motor mapea automáticamente los campos del nuevo contrato a los mecanismos internos existentes (planificador, motor de ventana temporal, motor de filtrado y motor de agregación), garantizando compatibilidad total con las métricas ya registradas.
Filtros avanzados: soporte de valores nulos
Nuevos operadores NULL y NOT_NULL para gestión explícita de nulos — KPIENGINE-1156

El modelo de filtros anterior no permitía gestionar valores nulos de forma explícita. Para excluir registros donde una dimensión era nula era necesario crear métricas auxiliares o duplicar lógica de filtrado. Los nuevos operadores eliminan esa necesidad.

IN
Incluye solo registros cuya dimensión esté en la lista de valores. Requiere values
NOT_IN
Excluye registros cuya dimensión esté en la lista. Requiere values
NULL
Incluye solo registros donde la dimensión es nula. No admite values
NOT_NULL
Excluye registros donde la dimensión es nula. No admite values
JSON — Ejemplo combinando NOT_IN y NOT_NULL
"filters": [
  {
    "aggregator": "CATEGORY_ID",
    "operator":   "NOT_IN",
    "values":     [123, 234, 345]
  },
  {
    "aggregator": "CATEGORY_ID",
    "operator":   "NOT_NULL"  // sin campo values
  }
]
⚠️
Regla de validación: si se incluye el campo values junto a los operadores NULL o NOT_NULL, el API responde con error 400. Los filtros son aditivos: un mismo KPI puede combinar múltiples operadores sobre la misma dimensión.
Migración del catálogo de métricas al nuevo contrato
Todas las métricas existentes se reexpresan bajo el nuevo modelo, de forma transparente para los consumidores

El objetivo es que todas las métricas del catálogo queden expresadas bajo el nuevo contrato, eliminando la coexistencia de dos modelos de configuración. Los valores calculados deben ser equivalentes a los del modelo anterior para el mismo rango temporal y granularidad.

📈 Métricas de ventas KPIENGINE-1125
Migración de todas las métricas de ventas (importe y unidades) con ventana DISCARD_AFTER_PROMOTION. El nuevo contrato representa cada ventana temporal como un campo window explícito, y la lógica de descarte de promociones se expresa en el campo schedule.
t_a24n t_u24n t_a72n t_u72n t_a168n …
🛒 Métricas de add-to-cart KPIENGINE-1140
16 métricas activas en producción, organizadas en tres grupos: sin agrupación por MOCACO, agrupadas por MOCACO en unidades, y agrupadas por MOCACO en importe. Se verificará el soporte del nuevo contrato para agregadores como CATEGORY_ID y SEGMENT_ID.
t_c4n t_c12n t_c24n t_c24pc t_ca24n t_ca24pcr …
📦 Métricas de stock KPIENGINE-1130
Las métricas de stock son de tipo snapshot (inventario actual, sin ventana deslizante). Se definirá cómo representar period.type: INFINITE en el nuevo contrato. Las métricas derivadas como ksmu24ni (ratio stock/ventas) requieren análisis adicional sobre soporte de fórmulas multi-fuente.
stock sm kstkcc ksmu24ni
📊 Ratios de conversión KPIENGINE-1154
Métricas compuestas que combinan dos fuentes (ventas e impresiones) mediante la fórmula unidades / (impresiones + 1). Se analizará si el nuevo contrato soporta fórmulas divisorias multi-fuente o si el ratio debe calcularse en el consumidor usando las métricas componentes.
kcrcs24n kcrcs10dn t_i24n t_i10dn
✅
Sin breaking change para consumidores: durante la transición ambos modelos coexisten. Los consumidores actuales siguen recibiendo los mismos valores. Una vez verificada la equivalencia en integración, el modelo anterior se retira.

Hoja de ruta
Secuencia de entrega para la adopción completa del nuevo modelo
Fase 01 · Fundación
Diseño e implementación del nuevo contrato de API (endpoint de creación y consulta) e integración con el flujo interno de cálculo. Validaciones, planificador, motor de filtros y agregación.
Fase 02 · Migración
Reexpresión de todo el catálogo existente bajo el nuevo contrato: ventas, add-to-cart, stock y ratios de conversión. Verificación de equivalencia en entorno de integración.
Fase 03 · Deprecación
Una vez completada la migración y validada la equivalencia, retirada del modelo anterior. Los consumidores habrán migrado su integración al nuevo contrato.
🔍 Puntos abiertos a resolver
  • ¿Cómo expresar métricas de tipo snapshot (sin ventana temporal deslizante) en el nuevo contrato? Las métricas de stock necesitan una definición documentada para window: INFINITE.
  • ¿Soporta el nuevo contrato fórmulas divisorias multi-fuente? Los ratios de conversión (kcrcs24n) combinan ventas e impresiones. Si no, el cálculo debe delegarse al consumidor.
  • ¿La proyección de COMMERCIAL_COMPONENT_ID en métricas por MOCACO se resuelve automáticamente por el motor o debe declararse explícitamente en aggregators?
  • ¿Se necesita un periodo de convivencia con ambos modelos activos en paralelo para garantizar cero impacto en consumidores actuales?
